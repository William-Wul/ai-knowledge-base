## 一句话抓重点

AI 圈最近冒出一股新声音，在反思过去两年"**模型越大越好、上下文窗口越长越好**"这条主路——走到后面发现，Agent 并不会因为模型变大、输入变长就真的"像人一样聪明"。**问题不在模型，在怎么组织喂给它的经验和记忆**。

## 放在一起看出来的事

### 共同的靶子：都在反对主流的"扁平 + 搜索"路径

这俩篇在批评**同一个东西**——当前 AI 圈处理经验和记忆的主流做法：

> 「把信息切成碎片 → 存进向量数据库 → 用户来查时按'最相似'捞几条 → 塞进上下文让模型回答」

这条路的两个代表性问题：

- **Gene 指出**：把 2500 字的完整 Skill 文档塞给模型，**不仅没帮助，反而比什么都不喂更糟**（在强模型 Gemini 3.1 Pro 上得分从 60 掉到 50）。原因是文档里写给人看的"背景介绍"对模型来说是干扰。
- **M-Flow 指出**：靠向量相似度捞片段，永远只能回答"**你问的这句话出现在哪里**"，但人脑真正值钱的能力是"**这事跟那事其实是一回事**"——那是联想，不是搜索。

### 共同的解法：结构化 + 关系化，而不是"更多"

两篇的技术方案不一样，但精神完全一致：

| | A002 的方案（Gene）| A003 的方案（M-Flow）|
|---|---|---|
| 核心动作 | 把经验**压缩**成策略片段 | 把记忆**分层**成情景/切面/事实 |
| 关键字段 | strategy + AVOID 清单 | Episode / Facet / FacetPoint + Entity 锚线 |
| 不追求什么 | 不追求"信息完整"  | 不追求"存得多" |
| 追求什么 | 控制密度——单位字数对模型的约束力 | 联想能力——能不能顺着关系网走 |

两个都不是在问"怎么让 AI 记得更多"，而是在问"**怎么让该被想起的东西在对的时刻被想起**"。

## 对你（William）的直接启示

### 1. 你对 skill 写太长的直觉是对的

你 2026-04-21 那天说"咱们上一个 skill 写得也挺长的"——这跟 Gene 论文的实验结论**完全一致**。我们当天就把 article-kb 这个 skill 从 6800 字压到 4900 字，砍掉了"用途介绍"这类给人看的段落。这不是跟风的优化，是有论文背书的正确方向。

### 2. "资料库存了但不看" 和 "Agent 只会搜不会联想" 是同一件事

你说的"**我可能也没有那么牛的记忆力和联想力去把资料库的东西更好的做联想和进一步思考**"——这正好对应 M-Flow 在批评的问题：**信息在那里，但不会自己浮现**。

这份主题笔记就是解法的人工版：我帮你把该连起来的东西连好，你不用翻原文也能吸收。未来可以继续扩展这个模式——不是 M-Flow 那种技术方案，但是用同样的思路（**让关系可见，让联想被代办**）。

### 3. 一个可以记一辈子的框架

两篇文章不约而同提到一对区分：

| | 能力指标 | 智能特质 |
|---|---|---|
| 代表 | "记得多 / 跑分高" | "该想到的时候能想到" |
| 特点 | 容易测，不稀缺 | 难测，真正值钱 |
| 例子（AI 侧）| benchmark 准确率 | 联想、灵感、迁移 |
| 例子（人/组织）| 员工知道多少 AI 术语 | 员工遇到新工具时的适应力 |

**这个框架可以直接迁移到你做 AI 学习知识库的目标判断上**：
- 浅目标：让员工记住更多名词 → 能力指标
- 深目标：让员工遇到新 AI 工具时不慌、能快速上手、能质疑 → 智能特质
- 前者容易做出来也容易测量；后者难做，但是真正能让公司受益的东西。

**写内容时值得时常自问**：我在做的这一篇，是在提升读者的能力指标，还是智能特质？如果大部分内容在做前者，要想想怎么多放一些后者的东西。
