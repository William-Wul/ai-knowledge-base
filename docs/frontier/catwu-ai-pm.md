# Claude产品负责人的PM方法论和AI时代工作方法

> 预计阅读时间：10 分钟 · 约 3000 字

---

Cat Wu 是 Anthropic Claude Code 和 Cowork 的产品负责人。

普林斯顿 CS 专业，做过工程师，去 Index Ventures 做了三年风险投资，2024 年 8 月加入 Anthropic。一年半后，她和 Claude Code 负责人 Boris Cherny 一起，把 Claude Code 做成了 Anthropic 300 亿美元年化收入里增长最猛的产品线。

最近她在 Lenny's Podcast 上聊了一个半小时。说了几件让人想认真思考的事。

![Cat Wu × Lenny Podcast 封面](/images/frontier-catwu-cover.png)

---

## 一、发布节奏从 6 个月压到 1 天，没写 PRD

先说一个数字上的对比。

过去的科技公司，一个功能从想法到用户能用到，通常要走 6 到 12 个月。PM（产品经理）大半时间花在写文档、对齐优先级、让所有人达成共识上。

Claude Code 现在的节奏：很多功能一周上线，有些甚至一天。

Anthropic 为了匹配这个节奏做了三件事，每一件都值得单拿出来看。

**① 把「发布」这件事的压力拆掉**

大多数功能发布时打上「研究预览」（Research Preview）标签——明确告诉用户这是早期产品，不保证长期维护。工程师有了新想法，一周内就能上线，不用等所有人达成共识再动。

**② 发布流水线分工**

工程师觉得功能可以了，往团队内部的「evergreen launch room」频道发一条消息。文档负责人、产品营销负责人第二天就把对外公告写好。整条链路里，工程师只负责把东西做出来，剩下的有人接。

**③ 用两份文档替代 PRD**

产品需求文档（PRD）这个东西没有消失，但只有两种情况才写：功能特别模糊，或者需要大量底层基础建设。其他所有情况，靠两份常驻文档兜底。

| 文档 | 频率 | 内容 |
|---|---|---|
| 每周全员数据复盘 | 每周一次，全员参加 | 业务关键指标、趋势、原因 |
| 团队原则文档 | 长期维护 | 核心用户是谁、什么重要、什么可以舍弃 |

Cat 说得很直接：「这两份东西一拿出来，团队里任何人都能自己做决策，不会卡在 PM 这边。」

## 二、为什么他们不招 PM，而是招工程师

外界一直在争论：AI 时代 PM 还需要不需要。

Cat 的回答绕开了这个问题本身，她说了一件更具体的事：

Claude Code 团队里，几乎所有 PM 都做过工程师，或者现在还在写代码。设计师也都是前端工程师出身。她自己也是从工程师转过来的。

「我们更愿意招有产品判断力的工程师，而不是再加一个 PM。」

理由很实际：这种人能从头到尾把活干完。在社区看见用户的抱怨，周五功能就能上线，整个过程不需要任何人介入。

但 Cat 也说了一句让人清醒的话：工程背景值不值钱，随着模型能力变化会反复翻转，她不敢说这个优势会一直在。

真正稳定值钱的东西，她管它叫「产品 taste」——就是那种能判断「这件事值不值得做、做成什么样才对」的直觉。

**「当写代码越来越容易的时候，决定写什么就变得越来越值钱。」**

这句话值得停一下想想。

## 三、AI 一晚上做完 20 页 PPT，但拍板的还是人

最近 Anthropic 有场 Code with Claude 开发者大会，Cat 要上台讲一节，主题是「Claude Code 怎么从 AI 助手变成一个完整的 Agent」。

她需要一份 PPT，里面要放 Anthropic 内部的真实使用案例。

她没有自己做。她用 Cowork（Anthropic 自己的 AI 助手产品）连接了公司的 Google Drive、Slack 和 Gmail，然后写了一段指令：

> 帮我做一份 Code with Claude 大会的 PPT。这是产品营销同事给的初版要点。这是我自己写的草稿，不满意，但放这儿了。能不能先给我一个详细的大纲。注意别和 keynote 主题撞车，keynote 优先级更高。

Cowork 干了一个小时。

它搜索了推特，看 Anthropic 最近上线了什么；查了 evergreen launch room，看团队在准备什么；又去翻了 Claude Code 的 demo 频道，看内部同事是怎么用自己产品的。

第二天早上，Cat 醒来收到一份 20 页的 PPT，套用了 Anthropic 内部的演讲模板和设计规范。

「读了一遍，挺不错的。文字稍微长了点，给了一轮反馈就改好了。」

她估算如果自己手做，至少要花几个小时。

![Cowork 自动生成 PPT 示意](/images/frontier-catwu-cowork.png)

但她补充了一句话，让这个案例没有变成「AI 替代人」的故事：

「Cowork 是非常好的头脑风暴助手。它能在几分钟内整合海量信息、把所有可能性列给你。**但最终留哪些，还得 PM 拍板。**」

## 四、一个没人在用的技巧：让 AI 解释自己的决定

Lenny 问 Cat：在 AI 时代，怎么培养那种「产品 taste」？

Cat 说了三件事，第一件最有意思，也最容易被忽视。

她说当 AI 做出让你意外的事，不要直接去调整设定——先问它「你刚才为什么这么决定」。

比如 Claude Code 改完一段前端代码，只跑了自动测试，没有验证界面实际长什么样。这在技术上说得过去，但不符合预期。

Cat 的做法不是直接改设定，而是问：「你刚才为什么这么决定？」

模型可能会说：「我的指令里这一段表述有歧义」，或者「我把验证这件事委派给了一个子程序，但我没有检查它是否真的执行了」。

「每次让它自我反思，你会立刻看到整个设定里哪里出了问题。这是优化提示词最快的办法。」

另外两件事：找 5 个反馈质量最高的真人（不需要大规模调研，一次小范围聚餐就够）；写 10 个可以量化的测试（把模糊的「感觉不对」翻译成具体的指标下降）。

## 五、给有缺陷的模型做产品，才是真正的难题

播客里 Cat 反复提到一个词：「AGI pilled」，意思是「相信 AGI（通用人工智能）真的快来了」的程度。

她说这件事是 PM 最难拿捏的。

她的逻辑是这样的：假设 AGI 明天就来了，你其实不需要复杂的产品——一个输入框就够了，什么都能做。真正难的不是给那个终极版模型设计产品。

「难的是给当下的模型设计：怎么把它现有的能力压榨到最大？怎么把用户引到它擅长的那条路上？怎么补足它的短板？」

她举了一个具体例子。早期 Claude Code 接到大型重构任务，改完 5 处就会停下来。团队就给它加了一个待办清单工具，强迫它把所有需要改的地方都列出来，挨个改完。

但 Opus 4 之后，这个问题消失了，模型自己就会维护待办清单。

「新模型出来时，要做的第一件事，往往是把加给上一代模型的那些辅助手段，一个个砍掉。」

这句话有一个更大的含义：AI 产品从来不是做完就完的东西。模型在升级，产品也要跟着升级——不是加功能，而是删掉那些为了弥补旧版本缺陷而加进去的东西。

## 六、一句话，送给所有觉得 AI 时代太快跟不上的人

播客结尾，Lenny 让 Cat 送一句话给焦虑的职场人。

她说了两个字：「Just do things.」（就去做事。）

她的意思是：职责边界是人为划定的，不是能力的上限。想清楚目标和约束，就放手去做，跨边界也行，做错了道歉再改回来。

但她也说了一种「错误的 just do things」：

把大量精力花在完善自己的工具、工作流上——加新插件、搭自动化流程、优化 AI 设定——结果真正要做的事情被搁置了。

「我也常看到有人把自动化做到 90%、95%，就放在那儿不管了。剩下那 5% 不啃下来，就根本算不上自动化。但那最后 5% 是真的费劲。」

她正在让 AI 帮她把 Gmail 收件箱清零，教了好几个月，还没到位。

「找你工作里那些重复的、你一点都不喜欢的活，把它交给 AI。然后用省下来的 20% 时间，去做你一直想干、但没有空干的事。」
